# 《逆序档案》开发路线

## 产品决定

- 核心品类：单人、PC 优先、买断制方向的卡牌肉鸽。
- 核心差异：玩家把行动、证据与改写牌编排到“过去 / 现在 / 未来”，再按时序统一结算。
- 单局目标：调查不断重演的零点灾难，以真相结案，同时管理跨案件保留的时间线与悖论。
- 当前基线：`nixu-dangan-demo/nixu-dangan-demo_9.html`，保留为行为与视觉参考，不再继续堆叠单文件版本。

## 阶段与完成标准

### 1. 基线审计与方向冻结

- 完整审阅原创设想、分析报告与 9 号原型。
- 冻结“因果时间轴”方向，放弃报告中的传统锋/守姿态方案。
- 记录现有内容规模、运行问题与工程债务。

### 2. 工程化重构

- 建立 Vite + React + TypeScript 工程。
- 内容数据、纯规则、存档迁移和界面分层。
- `pnpm test`、`pnpm typecheck`、`pnpm build` 全部通过。
- 保留旧原型存档的迁移入口。

### 3. 核心机制纵切

- 过去 / 现在 / 未来的选择必须对牌效和固定事件产生清晰差异。
- 支持撤回、能量返还、严格时序结算、战斗日志和可解释失败。
- 为卡牌、事件、遗物钩子和胜负条件建立规则单测。
- 完成一局 20–30 分钟的稳定闭环。

### 4. 构筑与关卡

- 扩展卡池、遗物、事件、普通案件、精英案件和 Boss。
- 建立约束式路线生成、固定种子回归和奖励反重复机制。
- 商店、休整、删牌、升级与轻元进度均能改变构筑策略。

### 5. 内容与叙事

- 三幕分别回答：灾难如何发生、谁在修改历史、玩家为何出现在第零号档案。
- 形成角色表、术语表、事件模板和文本审校规范。
- 分支信息影响玩家理解与结局，但不隐藏核心战斗规则。

### 6. 体验与表现

- 解决 1280px、移动窄屏和超宽屏的信息溢出。
- 完成分步教程、说明书、键盘操作、焦点状态、减弱动效和基础无障碍。
- 完成统一的卡面、节点、图标、动效、音乐和音效方向。

### 7. QA 与发布

- 规则单测、固定种子端到端回归、批量模拟和平衡阈值检查。
- 存档版本迁移、错误恢复、性能和发布构建检查。
- 产出可试玩构建、版本说明与已知问题清单。

## 设计门槛

新内容只有同时满足以下条件才进入正式池：

1. 位置有意义：放在不同时段至少出现两种可理解的结果。
2. 结果可解释：玩家能从卡面、固定事件和结算日志知道为什么发生。
3. 构筑有代价：强牌不能只比弱牌数值更大，必须改变资源、时序或风险。
4. 可回归：同一种子和同一决策序列必须得到同一结果。
